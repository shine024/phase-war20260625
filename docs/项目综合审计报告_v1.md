## [严重/CRITICAL] — 将阻止编译或导致崩溃

### C-001: 77个场景文件引用了不存在的脚本
**类别**: 场景完整性 / 编译失败
**影响**: 场景加载时会报 `Script "res://..." not found` 错误，相关场景无法运行

**详情**:
`scenes/main.tscn`, `scenes/title_screen.tscn`, 以及75个UI/单位/特效场景的脚本引用全部指向不存在的 `.gd` 文件。这些 `.gd` 文件实际上存在于项目其他位置（如 `scripts/ui/*.gd`），但场景文件中记录的 UID/path 与实际路径不匹配。

**根因**:
- 场景文件使用 UID 引用（`uid://...`），脚本文件可能被移动/重命名/删除后 UID 未更新
- Godot 编辑器中场景文件记录的脚本 UID 与实际文件丢失
- 这可能导致 Godot 编辑器无法加载这些场景

**解决方案**:
1. **方案A（推荐）**: 在 Godot 编辑器中打开每个场景，在 Inspector 中找到 `script` 属性，点击现有脚本重新关联
2. **方案B**: 编写脚本批量修复场景文件中的 ext_resource UID
3. **方案C**: 删除场景中的 script 引用，重新关联（会丢失 Godot 编辑器中可能存在的额外属性）

**优先级**: 立即处理 — 这是最高严重级别问题

### C-002: 26个 Autoload 引用了不存在的文件
**类别**: 项目配置 / 启动失败
**影响**: 项目启动时 autoload 列表全部标记为缺失

**详情**:
所有26个 autoload（SignalBus, GameManager, BattleManager, BlueprintManager 等）的文件路径在 `project.godot` 中注册，但文件扫描标记为 MISSING。这可能是因为：
- 文件路径中的 `*` 前缀是注释格式（如 `*res://scripts/signal_bus.gd`）
- 或者文件确实被移动/重命名

**注意**: 从输出看路径前缀带 `*`，这可能是 `project.godot` 中标记为"未加载/已禁用"的 autoload

**解决方案**:
1. 在 Godot 编辑器中打开 Project Settings → Autoload，检查每个 autoload 的实际状态
2. 如果文件存在但标记为禁用，重新选择文件启用
3. 如果文件确实丢失，从备份恢复或重新创建

**优先级**: 启动项目前必须解决

### M-003: 42个信号在多个文件中声明
**类别**: 信号重复
**影响**: 可能导致信号重复连接或冲突

**详情**:
- 大部分重复信号是 gdunit4 测试文件中的测试信号（无需处理）
- **项目信号重复**:
  - `achievement_unlocked` — achievement_manager.gd + signal_bus.gd
  - `blueprint_star_upgraded` — blueprint_manager.gd + signal_bus.gd
  - `card_obtained` — card_collection_manager.gd + signal_bus.gd
  - `challenge_started/completed/failed` — challenge_mode_manager.gd + signal_bus.gd
  - `quest_completed` — quest_manager.gd + signal_bus.gd
  - `story_chapter_started` — story_manager.gd + signal_bus.gd
  - `daily_tasks_refreshed` — daily_task_manager.gd + signal_bus.gd
  - `faction_reputation_changed` — faction_system_manager.gd + signal_bus.gd
  - `synthesis_completed/failed` — synthesis_manager.gd + signal_bus.gd
  - `closed` — 20+ 个 UI 面板声明（这是设计模式，每个面板关闭时发射）
  - `eom_unlocked` — enemy_origin_mod_manager.gd + intel_discovery_manager.gd
  - `eom_fragment_dropped` — enemy_origin_mod_manager.gd + intel_discovery_manager.gd

**分析**:
- 大部分信号在 manager 和 SignalBus 中同时声明是 **设计意图**（manager emit → SignalBus forward）
- 但如果 SignalBus 和 manager 都 `emit()` 同一个信号名，会导致重复处理
- `closed` 信号在每个UI面板中声明是合理的（每个面板独立事件）

**建议**: 确认 SignalBus 是否只是作为转发层（即 manager 发射后 SignalBus 再次发射），如果是，确保连接关系正确

**建议**: 检查 SignalBus 中是否有手动 `emit()` 调用转发 Manager 的信号
